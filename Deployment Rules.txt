# PTX Studio Deployment Rules

Last updated: 2026-05-09

This folder is the PTX Studio distribution layer.

It is deliberately separate from all development environments.

Nothing in this folder is experimental by default. Anything placed here should be treated as customer-facing or deployment-bound.

---

## Folder Roles

### ptx-download

`ptx-download` is the local mirror of the Cloudflare R2 downloads bucket.

It should reflect what exists, or is intended to exist, in:

`Cloudflare R2 bucket: ptx-downloads`

This folder controls customer download assets and download paths.

### ptx-website-deployment

`ptx-website-deployment` is the local deployable website package.

It should reflect the website files intended for Cloudflare deployment.

### R2_UPLOAD_ACCESS

`R2_UPLOAD_ACCESS` contains upload instructions, no-secret config examples, and helper scripts for future R2 uploads.

It must not contain real secrets, access keys, secret keys, exported credentials, or token screenshots.

### Deployment Rules.txt

This file defines the operating rules for this distribution layer.

If any instruction conflicts with this file, stop and ask Mario before continuing.

---

## Core Rule

Development builds must never be uploaded directly from a dev environment.

All release assets must pass through this `Deployments` folder first.

Correct flow:

1. Build in the dev environment.
2. Review and approve the build.
3. Copy the approved artifact into this `Deployments` folder.
4. Update manifests and download references here.
5. Verify local paths, names, versions, and file sizes.
6. Run an R2 dry-run report if using `rclone`.
7. Upload from this distribution layer only after approval.
8. Verify the live Cloudflare/R2 result.

Skipping this folder can cause customer-facing mistakes.

## Current Distribution Model

PTX Studio applications are currently distributed as portable standalone `.exe` files.

Customer-facing rule:

- No installer.
- No extraction.
- Download and launch.

Current primary download paths should use this model unless Mario explicitly changes the distribution strategy.

Expected current artifacts:

- `ptx-download\studio\latest\PTX Studio.exe`
- `ptx-download\editor_express\latest\PTX Editor Express.exe`
- `ptx-download\modules\editor_pro\latest\PTX Editor Pro.exe`

The current manifest files should describe these as:

- `type`: `standalone`
- `standalone`: `true`
- `install_type`: `portable`

Older installer, setup wizard, and `.7z` references may exist for historical reasons. Do not use them as current release guidance unless Mario confirms them.

## Manual Upload Rules

At this time, uploads are performed manually by Mario unless explicitly changed.

Before upload:

- Confirm the local file exists.
- Confirm the filename matches the public URL.
- Confirm the version number matches the release.
- Confirm the manifest URL matches the intended R2 object path.
- Confirm the website link points to the same object path.
- Confirm old installer/archive paths are not accidentally being promoted.

After upload:

- Confirm the object exists in R2.
- Confirm the public URL downloads the expected file.
- Confirm the file size is reasonable.
- Confirm the website CTA points to the correct download.
- Confirm the manifest is accessible if changed.

## R2/rclone Rules

When `rclone` is introduced, it must copy from this local folder only:

`D:\Dev Folder\PTX Studio\Deployments\ptx-download`

Target:

`Cloudflare R2 bucket: ptx-downloads`

Never copy or sync from:

- a dev environment
- `RAW`
- a temp build folder
- Downloads
- Desktop
- an unreviewed build output directory

Before any real `rclone` upload:

1. Run `R2_UPLOAD_ACCESS\r2-dry-run.ps1` first.
2. Review what will be uploaded or overwritten.
3. Do not use delete/sync-delete behavior unless Mario explicitly approves it for that run.
4. Confirm bucket target is `ptx-downloads`.
5. Confirm remote path is correct.
6. Confirm no unrelated files are included.
7. Get Mario's explicit approval.
8. Run `R2_UPLOAD_ACCESS\r2-upload-approved.ps1 -Approved` only after approval.

Default safe posture:

- Use `rclone copy`, not `rclone sync`.
- Upload/copy new or changed files.
- Avoid destructive delete operations.
- Avoid broad syncs until the workflow has been proven.

## Agent Rules

Agents may:

- Read this folder for deployment context.
- Audit manifests, paths, naming, and version consistency.
- Draft release checklists.
- Draft upload commands.
- Draft `rclone` dry-run commands.
- Explain what should be uploaded.
- Flag stale installer/archive references.
- Maintain `Deployment Rules.txt`.
- Maintain no-secret instructions and helper scripts inside `R2_UPLOAD_ACCESS`.

Agents must not:

- Upload files without exact founder approval.
- Delete files.
- Rename deployment artifacts.
- Modify website payload files unless explicitly authorized.
- Modify download payload files unless explicitly authorized.
- Run real `rclone` uploads without exact founder approval.
- Run Cloudflare deploy commands.
- Change DNS, workers, redirects, R2 settings, bucket settings, or access policy.
- Store real credentials in this folder.

Any action that changes customer-facing files or Cloudflare state requires explicit Mario approval for that exact action.

---

## Credential Rules

Real R2 credentials must not be stored in this folder.

Allowed:

- `rclone.conf.example` with placeholder values.
- Instructions explaining where to paste credentials during setup.
- Dry-run and upload scripts that refer to an existing configured remote.

Not allowed:

- Real Access Key ID.
- Real Secret Access Key.
- Cloudflare token screenshots.
- Exported environment files containing secrets.
- Plaintext credential backups.

Preferred credential location:

- Rclone's normal user config location.
- Rclone config encrypted with a config password, if available.
- A password manager controlled by Mario.

---

## Pre-Release Checklist

Before any release upload, verify:

- Product name is correct.
- Version is correct.
- File name is correct.
- File path is correct.
- Manifest path is correct.
- Website CTA path is correct.
- Release notes are ready if needed.
- License behavior is expected.
- Download file launches locally.
- No installer/setup language is being used unless intentionally restored.
- Backup or rollback path is known.

---

## Hard Stops

Stop immediately and ask Mario if:

- A manifest and actual file path disagree.
- A website link points to an older installer or archive.
- `latest` and `releases` disagree unexpectedly.
- The R2 bucket appears to contain files not present in this local mirror.
- A planned upload would overwrite or delete customer-facing assets unexpectedly.
- A file is larger than the browser upload limit and the S3/R2 workflow has not been approved.
- Any credential, token, or access key is visible.
- Any rule here conflicts with another document.

---

## Operating Principle

This folder exists to prevent mistakes.

The dev environment proves the product.

The deployment layer proves what customers receive.

Cloudflare should only receive files that have passed through this folder deliberately.
